# RFC Resource-Model Completion Design

**Date:** 2026-06-29
**Status:** Approved for implementation planning
**Scope:** `RFC-0101 — Serialization`, `RFC-0102 — Metadata Model`, `RFC-0103 — Resource Lifecycle`, `RFC-0105 — Manifest`, `RFC-0107 — Packaging Model`

---

## 1. Context

The `200-resource-model` cluster is partially complete: `RFC-0100` (Canonical Resource Model),
`RFC-0104` (Dependency Graph) and `RFC-0106` (Validation Model) are REAL. Five RFCs still contain
generic boilerplate:

- `RFC-0101 — Serialization`
- `RFC-0102 — Metadata Model`
- `RFC-0103 — Resource Lifecycle`
- `RFC-0105 — Manifest`
- `RFC-0107 — Packaging Model`

`RFC-0100` already forward-references all five (metadata `RFC-0102` eight times), `RFC-0106`
references `RFC-0101`/`RFC-0102`, and `RFC-0200` references `RFC-0107`. Completing these five
closes the largest remaining forward-reference group and finishes cluster `200` entirely (8/8).

## 2. Design Goal

Rewrite the five RFCs as one coherent resource-model completion cluster, each owning a distinct
facet of the canonical resource and delegating cross-cutting topics instead of re-defining them.
Remove boilerplate sections. Do NOT redefine the canonical envelope (`RFC-0100`), validation
diagnostics (`RFC-0106`), dependency graph (`RFC-0104`), resolver/lock/package mechanics, or
runtime flow.

## 3. Approved Approach

Use the **Resource-Model Completion** approach: per-RFC ownership split + a validator guard +
delegation via cross-reference (the proven foundation-rules pattern).

### 3.1 RFC-0101 — Serialization ownership
Owns: YAML and JSON as serialization *mappings* of the canonical object model; UTF-8 encoding;
deterministic serialization; round-trip fidelity; the rule that runtime engines operate on parsed
canonical objects, not raw text.
Required sections: `Format Policy`, `Encoding`, `Serialization Rules`, `Round-trip`, `Examples`, `References`.
Required rules:
- YAML and JSON are supported mappings; the canonical object model is the source of truth.
- Runtime engines MUST operate on parsed canonical objects, not raw YAML or JSON.
- Serialization SHOULD be deterministic; encoding is UTF-8.
- Field/envelope shape is owned by `RFC-0100`; metadata field semantics by `RFC-0102`.

### 3.2 RFC-0102 — Metadata Model ownership
Owns: the semantics of each metadata field used for identity, ownership, discovery, governance,
search and compatibility (`id`, `name`, `version`, `owner`, `labels`, `annotations`, and related
discovery/governance fields).
Required sections: `Metadata Fields`, `Identity`, `Discovery and Governance`, `Compatibility`, `Extensibility`, `Examples`, `References`.
Required rules:
- Metadata MUST include `id`, `name`, `version`.
- Identifier grammar for `id`/`name` is delegated to `RFC-0006`.
- Version format is delegated to `RFC-0005`; compatibility classification to `RFC-0007`.
- `labels` and `annotations` are extensible; unknown fields follow the active schema policy.
- The canonical envelope (where metadata sits) is owned by `RFC-0100`; lifecycle state by `RFC-0103`.

### 3.3 RFC-0103 — Resource Lifecycle ownership
Owns: lifecycle states and valid transitions.
Required sections: `Lifecycle States`, `State Transitions`, `Transition Rules`, `Status Field`, `Examples`, `References`.
Required rules:
- States are `Draft`, `Review`, `Approved`, `Published`, `Deprecated`, `Archived`.
- Transitions are directed; the normal path is `Draft → Review → Approved → Published → Deprecated → Archived`.
- `status.lifecycle` SHOULD be generated by tooling and MUST NOT define desired configuration (per `RFC-0100`).
- Governance/approval process is delegated to `RFC-0802`.

### 3.4 RFC-0105 — Manifest ownership
Owns: the structure of a manifest that declares/collects resources, packages and registries.
Required sections: `Manifest Purpose`, `Resource Manifest`, `Package Manifest`, `Registry Manifest`, `Manifest Rules`, `Examples`, `References`.
Required rules:
- A manifest declares or collects resources by `metadata.id`.
- A manifest MUST NOT redefine the canonical resource envelope (owned by `RFC-0100`).
- Package format/mechanics are delegated to `RFC-0107`/`RFC-0402`; registry mechanics to `RFC-0202`/`RFC-0403`.

### 3.5 RFC-0107 — Packaging Model ownership
Owns: the LOGICAL model of an immutable package (what a package is: contents, identity, immutability).
Required sections: `Package Concept`, `Package Contents`, `Package Identity`, `Immutability`, `Examples`, `References`.
Required rules:
- Packages are immutable once produced.
- A package is identified by name + version + checksum (checksum/signing mechanics delegated to `RFC-0801`).
- This RFC defines the logical model only; compression/signing/verification mechanics are delegated to `RFC-0402`, publishing to `RFC-0403`, and the artifact relationship to `RFC-0207`.

## 4. Out of Scope
- Canonical resource envelope fields; owned by `RFC-0100`.
- Dependency graph semantics; owned by `RFC-0104`.
- Validation layers and diagnostic envelope; owned by `RFC-0106`.
- Version format and ranges (`RFC-0005`), naming grammar (`RFC-0006`), compatibility classification (`RFC-0007`).
- Resolver/version selection (`RFC-0204`), lock snapshots (`RFC-0401`).
- Packaging mechanics (`RFC-0402`), registry/marketplace (`RFC-0403`), signing/integrity (`RFC-0801`).
- Runtime flow; owned by `RFC-0200`.

## 5. Validator Design
Extend `tools/check-rfc-links` with a resource-model guard after the foundation-rules guard.
The guard should:
- Verify header fields in `RFC-0101`, `RFC-0102`, `RFC-0103`, `RFC-0105`, `RFC-0107`.
- Reject boilerplate headings (`Motivation`, `Goals`, `Non-Goals`, `Canonical Model`, `Required Behavior`, `Runtime Flow`, `Validation Rules`, `Error Model`, `Security Considerations`, `Migration Guidance`, `Future Work`).
- Require each RFC's section set listed in §3.
- Require ownership-proving key phrases:
  - `RFC-0101`: `parsed canonical` and `YAML and JSON`.
  - `RFC-0102`: delegates identifier grammar to `RFC-0006` and version format to `RFC-0005`.
  - `RFC-0103`: contains all six states (`Draft` … `Archived`) and delegates governance to `RFC-0802`.
  - `RFC-0105`: `manifest` and delegates package to `RFC-0107`/`RFC-0402`.
  - `RFC-0107`: `immutable` and delegates mechanics to `RFC-0402`/`RFC-0801`.
- Use exact-heading checks where possible to avoid false positives in prose/references.

## 6. Implementation Plan Shape
Split into six commits, preserving fail-first evidence and keeping each RFC reviewable:
1. Add resource-model validator guard and verify it fails against current boilerplate.
2. Rewrite `RFC-0101 — Serialization`; verify the guard now fails only for the remaining RFCs.
3. Rewrite `RFC-0102 — Metadata Model`.
4. Rewrite `RFC-0103 — Resource Lifecycle`.
5. Rewrite `RFC-0105 — Manifest`.
6. Rewrite `RFC-0107 — Packaging Model`; run full `tools/check-rfc-links`; if `TODO.md` carries a
   pending-delegation entry covering these RFCs, close it.

## 7. Acceptance Criteria
- `RFC-0101`, `RFC-0102`, `RFC-0103`, `RFC-0105`, `RFC-0107` no longer contain boilerplate ownership sections.
- Each RFC has its required sections and references owner RFCs instead of duplicating their scope.
- `RFC-0102` does not redefine the canonical envelope; `RFC-0107` stays logical-only (no packaging mechanics).
- `tools/check-rfc-links` passes after the cluster is implemented.
- `SUMMARY.md` links remain unchanged.
- Cluster `200-resource-model` is fully REAL (8/8 RFCs).

## 8. Recommendation
Proceed with this cluster as the next implementation cycle. It finishes the resource-model layer
end-to-end, closing `RFC-0100`'s forward-reference group while leaving resolver, build-distribution
and security mechanics for later RFC-specific cycles.
