# Design — RFC-0200 Runtime Architecture

- **Ngày:** 2026-06-29
- **Phạm vi:** Phase A (hoàn thiện đặc tả), sub-project — `docs/300-runtime/RFC-0200-Runtime-Architecture.md`
- **Approach:** B — Engine boundary contract
- **Trạng thái:** Draft (chờ review)

## Bối cảnh

`RFC-0200` được `RFC-0003` và `TODO.md` xem là nguồn sở hữu Runtime Flow. Hiện RFC này vẫn còn boilerplate chung §2–§14, trong đó runtime flow chỉ là một sơ đồ tuyến tính lặp lại ở nhiều RFC khác. Sau khi đã hoàn thiện `RFC-0100` (resource envelope) và `RFC-0106` (validation result/error model), `RFC-0200` cần chốt pipeline runtime tổng và boundary giữa các engine để các RFC engine con `RFC-0201..0209` có điểm tựa.

## Mục Tiêu

- Định nghĩa runtime pipeline tổng của ARPS.
- Chốt trách nhiệm và boundary của từng engine runtime.
- Định nghĩa handoff artifacts giữa các engine ở mức kiến trúc.
- Chốt runtime invariants như không execution trước validation pass và không mutate source resources trong planning/read-only phases.
- Loại bỏ boilerplate không thuộc trách nhiệm của `RFC-0200`.

## Không Làm

- Không định nghĩa thuật toán chi tiết của từng engine; phần đó thuộc `RFC-0201..0209`.
- Không định nghĩa validation result shape; phần đó thuộc `RFC-0106`.
- Không định nghĩa dependency graph model chi tiết; phần đó thuộc `RFC-0104`.
- Không định nghĩa cache strategy; phần đó thuộc `RFC-0208`.
- Không định nghĩa observability signals chi tiết; phần đó thuộc `RFC-0209`.
- Không định nghĩa build/package registry protocol chi tiết; phần đó thuộc `RFC-0400..0403`.

## Thiết Kế Đề Xuất

`RFC-0200` sẽ được viết lại thành RFC kiến trúc runtime sở hữu pipeline tổng, engine boundary và handoff artifacts, giữ header chuẩn đã chốt ở ADR-0003.

### 1. Abstract

Tóm tắt rằng RFC này định nghĩa runtime architecture và engine boundaries cho ARPS.

### 2. Conventions

Nêu quy ước RFC 2119 và ownership boundary:
- Resource envelope: `RFC-0100`.
- Dependency graph: `RFC-0104`.
- Validation result/error model: `RFC-0106`.
- Packaging model: `RFC-0107`.
- Engine-specific behavior: `RFC-0201..0209`.
- Build/distribution/registry publishing: `RFC-0400..0403`.

### 3. Runtime Pipeline

Pipeline tổng:

```text
Repository
  -> Discovery Engine
  -> Registry Engine
  -> Validation Engine
  -> Dependency Resolver
  -> Planning Engine
  -> Execution Engine
  -> Packaging Engine
  -> Publishing Engine
  -> Registry / Marketplace / Consumer
```

Pipeline này là thứ tự logical handoff. RFC-0200 không bắt buộc implementation phải chạy mọi bước trong một process hoặc luôn chạy tuần tự vật lý, miễn handoff contract và invariants được giữ.

### 4. Engine Boundary Matrix

| Engine | Input | Output | Owns |
|---|---|---|---|
| Discovery Engine | repository/filesystem/package source | `ResourceCandidate[]` | locating candidate resource files and manifests |
| Registry Engine | resource candidates | `CanonicalResourceSet` / registry index | canonical indexing, identity lookup, registry view |
| Validation Engine | canonical resources + schema/policy context | `ValidationResult` | validation execution using `RFC-0106` result model |
| Dependency Resolver | validated resources + registry index | `DependencyGraph` / resolution set | dependency graph resolution and ordering |
| Planning Engine | resolution set + user intent | `ExecutionPlan` | deterministic plan construction |
| Execution Engine | execution plan + adapters/tools | `ExecutionArtifact[]` / execution result | invoking declared adapters/capabilities |
| Packaging Engine | artifacts + resources + metadata | `PackageBundle` | producing distributable package bundle |
| Publishing Engine | package bundle + registry target | `PublishResult` | publishing, registry update, marketplace handoff |

### 5. Handoff Artifacts

Define artifacts at architecture level:
- `ResourceCandidate`: discovered source path plus minimal parse/discovery metadata.
- `CanonicalResourceSet`: normalized set of resources keyed by registry identity.
- `ValidationResult`: validation output defined by `RFC-0106`.
- `DependencyGraph`: graph of resource dependency nodes/edges defined by `RFC-0104`.
- `ExecutionPlan`: deterministic ordered plan produced before execution.
- `ExecutionArtifact`: result generated by execution adapters/tools.
- `PackageBundle`: package output defined by packaging/build RFCs.
- `PublishResult`: result of publishing to registry/marketplace target.

### 6. Runtime Invariants

- Execution MUST NOT start before required validation has passed.
- Dependency resolution MUST use validated resources.
- Planning MUST NOT mutate source resource files.
- Execution MUST use declared adapters/capabilities, not implicit vendor-specific behavior.
- Packaging MUST use validated resource metadata and produced artifacts.
- Publishing MUST operate on a package bundle, not raw unchecked resources.
- Given identical inputs and context, handoff artifacts SHOULD be deterministic.

### 7. Failure Boundaries

- Discovery failures produce discovery diagnostics or engine-specific results owned by `RFC-0201`.
- Validation failures use `ValidationResult` from `RFC-0106`.
- Compatibility conflicts use classification from `RFC-0007` and diagnostics from `RFC-0106`.
- Dependency failures are owned by `RFC-0104`/`RFC-0204` and surfaced before planning.
- Execution/package/publish failures are owned by their engine RFCs and build/distribution RFCs.

### 8. Examples

RFC nên có ví dụ ngắn:
- Happy path: repository resources → validation pass → graph resolved → plan → artifacts → package → publish.
- Stop at validation: invalid metadata produces `ValidationResult.valid=false`; resolver/planner/execution do not run.
- Stop at dependency: validation passes but resolver reports missing dependency; planning/execution do not run.

## Ranh Giới Với Các RFC Khác

| Chủ đề | RFC sở hữu | Cách `RFC-0200` xử lý |
|---|---|---|
| Resource envelope | `RFC-0100` | Dùng làm input/output resource contract. |
| Validation result | `RFC-0106` | Chỉ nêu Validation Engine trả `ValidationResult`. |
| Dependency graph | `RFC-0104` | Chỉ nêu resolver tạo graph/resolution set. |
| Engine internals | `RFC-0201..0209` | Chỉ định nghĩa boundary, không định nghĩa thuật toán. |
| Cache/incremental | `RFC-0208` | Không chốt trong RFC-0200. |
| Observability | `RFC-0209` | Không chốt signal/log/metric chi tiết. |
| Build/publish protocol | `RFC-0400..0403` | Chỉ nêu packaging/publishing boundary. |

## Acceptance Criteria Cho Vòng Implementation Sau

- `docs/300-runtime/RFC-0200-Runtime-Architecture.md` không còn boilerplate chung §2–§14.
- RFC có các section riêng: `Runtime Pipeline`, `Engine Boundary Matrix`, `Handoff Artifacts`, `Runtime Invariants`, `Failure Boundaries`, `Examples`, `References`.
- RFC chốt thứ tự logical runtime flow từ Discovery tới Publishing.
- RFC nêu rõ engine boundaries và handoff artifacts.
- RFC nêu rõ execution không chạy trước validation pass.
- RFC cross-ref đúng các RFC sở hữu resource model, validation model, dependency graph, engine con, build/package/registry.
- `bash tools/check-rfc-links` vẫn PASS; nếu cần rule riêng cho RFC-0200 thì thêm trong implementation plan, không tự làm trong design này.

## Rủi Ro Và Giảm Thiểu

- **Lấn sang RFC-0201..0209:** chỉ định nghĩa boundary và handoff, không định nghĩa thuật toán engine.
- **Lấn sang RFC-0208:** không chốt cache/incremental/parallel semantics.
- **Lấn sang RFC-0106:** chỉ dùng `ValidationResult`, không định nghĩa lại diagnostic model.
- **Pipeline quá cứng:** mô tả thứ tự logical handoff, không ép physical execution phải luôn tuần tự.
