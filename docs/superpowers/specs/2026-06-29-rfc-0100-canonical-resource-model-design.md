# Design — RFC-0100 Canonical Resource Model

- **Ngày:** 2026-06-29
- **Phạm vi:** Phase A (hoàn thiện đặc tả), sub-project — `docs/200-resource-model/RFC-0100-Canonical-Resource-Model.md`
- **Approach:** B — Envelope + invariants
- **Trạng thái:** Draft (chờ review)

## Bối cảnh

`RFC-0100` được `RFC-0000`, `RFC-0007` và nhiều tài liệu nền xem như nguồn sở hữu canonical resource model. Hiện RFC này vẫn còn cấu trúc boilerplate chung §2–§14, nhưng đã có một mảnh contract riêng ở cuối:

> `apiVersion`, `kind`, `metadata` and `spec` are required. `status` SHOULD be generated by tooling and MUST NOT define desired configuration.

Mục tiêu vòng này là biến mảnh contract đó thành nội dung chính của RFC: định nghĩa resource envelope chung cho mọi Resource Kind, cùng semantics và invariants đủ rõ để các RFC khác cross-ref.

## Mục Tiêu

- Định nghĩa canonical resource envelope cho ARPS.
- Chốt ý nghĩa và trách nhiệm của 5 top-level fields: `apiVersion`, `kind`, `metadata`, `spec`, `status`.
- Chốt required behavior tối thiểu cho producer, parser, consumer và read-only phases.
- Tách rõ `spec` là desired/authored intent và `status` là observed/generated state.
- Loại bỏ boilerplate không thuộc trách nhiệm của `RFC-0100`.

## Không Làm

- Không định nghĩa chi tiết metadata fields; phần đó thuộc `RFC-0102 Metadata Model`.
- Không định nghĩa lifecycle state machine; phần đó thuộc `RFC-0103 Resource Lifecycle`.
- Không định nghĩa dependency graph; phần đó thuộc `RFC-0104 Dependency Graph`.
- Không định nghĩa manifest packaging; phần đó thuộc `RFC-0105 Manifest` và `RFC-0107 Packaging Model`.
- Không định nghĩa validation rule taxonomy hoặc error output; phần đó thuộc `RFC-0106 Validation Model`.
- Không định nghĩa compatibility classification; phần đó thuộc `RFC-0007 Compatibility Policy`.

## Thiết Kế Đề Xuất

`RFC-0100` sẽ được viết lại thành RFC sở hữu top-level resource envelope và invariant chung, giữ header chuẩn đã chốt ở ADR-0003.

### 1. Abstract

Tóm tắt rằng RFC này định nghĩa canonical envelope dùng chung cho mọi ARPS Resource Kind.

### 2. Conventions

Nêu quy ước RFC 2119 và ownership boundary:
- Naming/resource kind conventions: `RFC-0006`.
- Serialization: `RFC-0101`.
- Metadata chi tiết: `RFC-0102`.
- Lifecycle chi tiết: `RFC-0103`.
- Manifest/packaging: `RFC-0105`, `RFC-0107`.
- Validation/errors: `RFC-0106`.
- Compatibility: `RFC-0007`.

### 3. Resource Envelope

Đưa YAML mẫu tối thiểu:

```yaml
apiVersion: platform/v1
kind: Skill
metadata:
  id: core/clean-code
  name: clean-code
  version: 1.0.0
spec: {}
```

Và YAML có generated status:

```yaml
apiVersion: platform/v1
kind: Skill
metadata:
  id: core/clean-code
  name: clean-code
  version: 1.0.0
spec: {}
status:
  observedGeneration: 1
  lifecycle: Published
```

Top-level required fields:
- `apiVersion`
- `kind`
- `metadata`
- `spec`

Top-level optional/generated field:
- `status`

### 4. Top-Level Fields

| Field | Required | Owner | Semantics |
|---|---|---|---|
| `apiVersion` | Yes | `RFC-0100` + serialization/schema family | API/spec family used to interpret the resource envelope. |
| `kind` | Yes | `RFC-0006` + kind RFC | PascalCase Resource Kind selecting the resource contract. |
| `metadata` | Yes | `RFC-0102` | Identity, version and registry/search metadata. |
| `spec` | Yes | kind-specific RFC | Desired configuration, capability declaration or authored intent. |
| `status` | No | runtime/tooling + lifecycle RFC | Observed/generated state; not desired configuration. |

### 5. Required Behavior

- Producers MUST emit `apiVersion`, `kind`, `metadata` and `spec`.
- Producers MAY omit `status`.
- Consumers MUST NOT require `status` for initial parse, validation or dependency resolution.
- Read-only phases MUST NOT mutate source resource files.
- Tools that generate `status` MUST keep desired configuration in `spec`, not `status`.
- Parsers SHOULD preserve known fields and SHOULD NOT drop unknown extension fields unless the active schema policy requires rejection.
- Unknown top-level fields are handled by schema/validation policy, not ad-hoc per implementation.

### 6. Envelope Invariants

- `apiVersion` + `kind` select the schema family for interpreting `spec`.
- `metadata.id` identifies the resource inside a registry context.
- `metadata.version` identifies the resource instance version; it is distinct from `apiVersion`.
- `spec` represents authored desired state or declared capability.
- `status` represents observed/generated state and MUST NOT be required to reproduce desired behavior.
- `metadata` MUST NOT be used as a substitute for kind-specific behavioral configuration.

### 7. Examples

RFC nên có ví dụ ngắn:
- Minimal valid resource.
- Resource with labels/annotations and empty status omitted.
- Resource with generated status.
- Invalid pattern: desired config stored in `status`.

## Ranh Giới Với Các RFC Khác

| Chủ đề | RFC sở hữu | Cách `RFC-0100` xử lý |
|---|---|---|
| Naming/id/kind convention | `RFC-0006` | Chỉ yêu cầu `kind` và `metadata.id`, không định nghĩa grammar chi tiết. |
| Metadata fields | `RFC-0102` | Chỉ nói metadata là identity/version/search metadata. |
| Lifecycle/status values | `RFC-0103` | Chỉ nói `status` là observed/generated state. |
| Serialization/YAML/JSON | `RFC-0101` | Chỉ dùng YAML ví dụ, không định nghĩa serializer. |
| Validation errors | `RFC-0106` | Chỉ nêu required fields và invariant; error/reporting thuộc RFC-0106. |
| Compatibility | `RFC-0007` | Chỉ cross-ref khi envelope thay đổi. |

## Acceptance Criteria Cho Vòng Implementation Sau

- `docs/200-resource-model/RFC-0100-Canonical-Resource-Model.md` không còn boilerplate chung §2–§14.
- RFC có các section riêng: `Resource Envelope`, `Top-Level Fields`, `Required Behavior`, `Envelope Invariants`, `Examples`, `References`.
- RFC chốt required fields: `apiVersion`, `kind`, `metadata`, `spec`.
- RFC chốt `status` là optional/generated và không chứa desired configuration.
- RFC cross-ref đúng các RFC sở hữu metadata, lifecycle, validation, compatibility và serialization.
- Không sửa các RFC liên quan trong cùng task, trừ khi cần cập nhật validator hoặc TODO pending.
- `bash tools/check-rfc-links` vẫn PASS; nếu cần rule riêng cho RFC-0100 thì thêm trong implementation plan, không tự làm trong design này.

## Rủi Ro Và Giảm Thiểu

- **Lấn sang RFC-0102:** chỉ mô tả vai trò metadata, không liệt kê/chuẩn hóa sâu từng metadata field.
- **Lấn sang RFC-0103:** chỉ mô tả `status` là observed state, không định nghĩa lifecycle state machine.
- **Lấn sang RFC-0106:** validation output/error taxonomy chỉ forward-ref.
- **Schema quá cụ thể:** RFC-0100 giữ contract envelope; JSON/YAML schema chi tiết có thể được đồng bộ ở vòng schema/tooling sau.
